WMAutoattack Compare-Search Notes

Purpose
This file explains the added "Claudini-style vs ours" comparison code in this repository,
so a Codex instance on another machine/server can quickly understand what was changed
and how to run the experiments.

High-level idea
The repository now contains a fair comparison entrypoint for two LLM-based attack-search methods:

1. claudini_style
   A generic LLM self-evolving attack-search baseline inspired by Claudini.
   In this repo it is implemented as:
   - same search space
   - same evaluator
   - same trial budget
   - same LLM backend
   - no task-specific retrieval by default
   This is NOT the original Claudini repo. It is a Claudini-style baseline implemented locally.

2. ours
   The method under comparison.
   In the current setup it uses:
   - same attack search space
   - same evaluator
   - same budget
   - same LLM backend
   - latent retrieval enabled by default

Nothing in this comparison path imports or depends on the separate Claudini repository.
Everything runs entirely inside D:\WMAutoattack.

Files added or modified
1. agent\compare_search.py
   New comparison entrypoint.
   Responsibilities:
   - build shared baseline/task profile once
   - create method-specific configs
   - seed isolated memory stores per method
   - optionally exclude same-checkpoint memory entries
   - run claudini_style and ours under the same protocol
   - write per-method summaries and final comparison summary

2. agent\orchestration.py
   Modified so the controller can reuse an externally computed baseline_result and task_profile.
   This avoids recomputing a separate clean baseline per method and keeps the comparison fair.

3. main.py
   Added new CLI command:
   - compare-search

4. tests\integration\test_main_entrypoints.py
   Added help/entrypoint test for compare-search.

5. tests\agent\test_compare_search.py
   Added a test that verifies the seeded memory store excludes the target checkpoint when requested.

Main command
The new CLI command is:

python main.py compare-search [args...]

You can also inspect the CLI help with:

py -3.10 main.py compare-search --help

How compare-search works
Step 1. Discover tasks/checkpoints.
It accepts either:
- --checkpoint-path <ckpt1> [ckpt2 ...]
or
- --task-root plus discovery settings

Step 2. Build a shared clean baseline.
It runs a single baseline evaluation for each task and derives a shared task_profile.
This shared baseline is reused by both methods.

Step 3. Create two method-specific search runs.
By default:
- baseline method name: claudini_style
- baseline retrieval mode: none
- ours method name: ours
- ours retrieval mode: latent

Step 4. Copy and isolate the experience store per method.
Each method receives its own local experience_store.jsonl inside its output directory.
By default, entries from the exact target checkpoint are filtered out to avoid leakage.

Step 5. Run both methods under the same settings.
Both methods share:
- same attack space
- same runtime budget
- same trial budget
- same scout/confirm episode counts
- same LLM backend/model
- same DreamerV3 evaluator

Step 6. Write summaries.
Outputs include:
- per-method search_summary.json
- per-method search_summary.txt
- global comparison_summary.json
- global comparison_summary.txt

Important fairness settings
The intended default fair setup is:
- claudini_style uses --baseline-retrieval-mode none
- ours uses --ours-retrieval-mode latent
- same task checkpoint
- same shared baseline
- same max_trials_per_attack
- same attacks list
- same LLM backend/model
- same scout/confirm protocol
- same accelerator/devices/precision

Important note:
If you allow same-checkpoint memory, the comparison may become biased.
Default behavior is safer because compare_search excludes identical checkpoint entries
from the seeded memory store unless --allow-same-checkpoint-memory is explicitly passed.

GPU notes
To run on GPU, the environment must have CUDA-enabled PyTorch.
The working local environment used during testing was:
- Python 3.10
- torch 2.6.0+cu124
- torchvision 0.21.0+cu124
- torchaudio 2.6.0+cu124

Important:
If torch is CPU-only, compare-search will not use CUDA even if you pass:
- --accelerator cuda

Minimal CUDA verification:

py -3.10 - <<'PY'
import torch
print(torch.__version__)
print(torch.cuda.is_available())
print(torch.version.cuda)
if torch.cuda.is_available():
    print(torch.cuda.get_device_name(0))
PY

Expected successful state:
- torch.cuda.is_available() == True

Server-side checklist
When syncing to a server, verify the following before running:

1. Python version matches the environment used by the repo.
2. PyTorch is CUDA-enabled if GPU execution is desired.
3. The server has access to the required checkpoints.
4. The server has access to the Hugging Face model cache or network download access.
5. The server has the optional dependencies needed by this repo:
   - atari extras
   - local-llm extras
6. The server uses the correct working directory:
   - D:\WMAutoattack or the synced equivalent path

Suggested install sequence on server
Example:

py -3.10 -m pip install -e .[atari,llm,local-llm]

Then ensure CUDA torch is correct.
If needed, reinstall official CUDA wheels directly.

Interpretation of current setup
At the moment, the baseline is:
- generic LLM self-evolution without retrieval

The proposed method is:
- LLM self-evolution with latent retrieval

If the experience store is sparse, the advantage of latent retrieval may be small or unstable.
For stronger experiments, populate logs\agent_experience.jsonl with more relevant prior tasks.

What this code does NOT do
- It does not import the external Claudini repository.
- It does not run token-suffix attacks from the LLM benchmark repo.
- It does not compare against the original Claudini implementation directly.

It only compares:
- a Claudini-style local baseline
vs
- the local latent-RAG method

Quick summary for server Codex
If you are another Codex instance reading this file:

1. Work inside WMAutoattack, not the external Claudini repo.
2. Use main.py compare-search as the entrypoint.
3. Keep claudini_style = retrieval none.
4. Keep ours = retrieval latent.
5. Use the same checkpoint, same attacks, same budgets, same LLM.
6. Prefer CUDA if available.
7. Read comparison_summary.txt first after each run.
